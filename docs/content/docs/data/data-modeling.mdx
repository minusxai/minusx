---
title: Data Modeling
description: A trusted, governed layer over your raw tables — query with no SQL, or curate SQL into clean tables
---

Data modeling in MinusX is how you turn raw warehouse tables into a **trusted layer** your team — and the AI — can query with confidence. Instead of everyone re-deriving the same joins and filters, the logic lives in one place.

There are two complementary pieces, and neither copies or materializes data into your warehouse — models are **derived on demand** and Views are **virtual**:

- **Semantic models** power a no-code **GUI** for asking measures-by-dimensions questions.
- **Views** let you curate SQL into a clean, flat table when a question needs a shape the semantic layer can't express.

Both are configured in the [Knowledge Base](/docs/knowledge-base/context).

---

## Semantic models & the GUI

For every table (and View) you whitelist, MinusX automatically derives a **semantic model** — a business-friendly description of what you can ask:

- **Numeric** columns become **measures** (sum, average, count)
- **Categorical / text** columns become **dimensions** you group by
- **Date / time** columns become a **time axis** with selectable grain (day, week, month…)
- The **relationships** you define in the Knowledge Base become the **joins**

With a model in place, the **GUI** tab lets anyone build a query by picking measures, dimensions, filters, and a time grain — **no SQL required**. The GUI, SQL, and Viz tabs are three views of the *same* query, so you can start in the GUI and drop to SQL (or the reverse) at any time.

<ThemedImage light="/light/semantic-gui.png" dark="/dark/semantic-gui.png" width={1200} height={637} alt="The GUI query builder — pick a model, measures and dimensions, and get a chart with no SQL" />

<Callout type="info">
The GUI expresses **aggregate** questions — one or more measures grouped by dimensions. A raw row dump like `SELECT *` isn't a semantic query, so the GUI stays disabled for it (switch to the SQL tab for those). This is expected, not an error.
</Callout>

Models are derived on the fly and scoped to the tables a query actually touches — nothing is stored or materialized. To make the GUI smarter, define your **relationships** (so joins are correct) and **metrics** (for ratios like conversion rate) in the Knowledge Base.

## Views

Sometimes a question needs a shape the semantic layer can't reach — a specific multi-table join, a union, a window function. A **View** lets you write that SQL once and expose the result as a **clean, flat table**.

<ThemedImage light="/light/views.png" dark="/dark/views.png" width={1200} height={637} alt="A View in the Knowledge Base — the definition editor open below the whitelist row, with a live preview of the result" />

- **Author like a question.** You build a View in the Knowledge Base using the same editor as any question — GUI / SQL / Viz, with Run and a live preview — then save it. It appears under a `_views` schema and behaves like any other table: columns, types, and it can even back the GUI.
- **Virtual, not materialized.** MinusX inlines a View's SQL into your query at run time. Nothing is written to your warehouse, so Views work on **read-only** connections and across **every** SQL dialect.
- **Whitelist a View like a table — down to the column.** Expose the whole View, or tick individual columns. Hiding a column is real enforcement: it disappears from the View's output for everyone — the AI, the GUI, and every query — not just visually.
- **Compose them.** A View can reference other Views, so you can layer curated logic.

Views obey the **same governance tree** as table whitelisting. A View may read any table the context's *parent* offers — even one you've curated away from end users — but never a table outside the whitelist chain. If a parent later narrows what it offers, any dependent View is **disabled with a reason** rather than silently reading data that was pulled.

<Callout type="info">
Reach for the **semantic layer** first for standard measures-by-dimensions questions. Reach for a **View** when you need a specific join or shape the semantic model can't express — it's the escape hatch, not the default.
</Callout>

## Where they live & best practices

Both are configured in the **Knowledge Base → context → Databases** tab. Semantic models are derived from the tables and Views you whitelist there; Views are created and column-whitelisted in the **Views** section of the same panel.

- **Model to a clear grain** — decide what one row of a View represents before you write it (one order, one zone-day).
- **Keep Views flat and self-contained** — a business user should query the View without joining anything else.
- **Name for intent** — `zone_revenue`, `active_subscriptions` read better than `v1`.
- **Define relationships and metrics** so the GUI joins correctly and computes ratios the way your team means them.
- **Document the non-obvious** in [text context](/docs/knowledge-base/context) — the AI reads it on every query.
